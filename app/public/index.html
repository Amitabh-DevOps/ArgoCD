Foundation

What is GitOps?

GitOps is an operational framework that takes DevOps best practices — version control, collaboration, compliance, CI/CD — and applies them to infrastructure automation.

Coined by Weaveworks in 2017, GitOps uses Git repositories as the single source of truth for defining the desired state of your infrastructure and applications. Instead of running imperative commands (kubectl apply), you push changes to Git — and an automated operator reconciles the live system to match.

This means every change is versioned, auditable, and reversible. Rollbacks become as simple as git revert.

The Tool

What is ArgoCD?

ArgoCD is a declarative, GitOps continuous delivery tool for Kubernetes — a CNCF Graduated project.

📋

Declarative Configuration

Define what you want, not how to get there. ArgoCD reads your Kubernetes manifests, Helm charts, or Kustomize configs from Git and ensures the cluster matches that state.

🔄

Automated Sync

ArgoCD continuously monitors both Git and the cluster. When drift is detected — a manual change or a new commit — it automatically (or on-demand) reconciles the cluster back to the desired state.

🖥️

Rich UI & CLI

A real-time web dashboard visualises application health, sync status, resource trees, and deployment history. Full CLI support for automation and scripting.

🌐

Multi-Cluster Management

Manage deployments across many Kubernetes clusters from a single ArgoCD control plane. Ideal for staging → production promotion workflows.

🔐

RBAC & SSO

Fine-grained role-based access control with support for OIDC, GitHub, GitLab, LDAP, and SAML SSO integrations out of the box.

⏪

Instant Rollback

Every deployment is tied to a Git commit. Rolling back is as simple as reverting the commit or pointing ArgoCD to a previous revision — no runbooks needed.

Core GitOps Principles

The Four Principles

OpenGitOps (CNCF) defines four foundational principles that every GitOps implementation must follow.

01

Declarative

The entire system must be described declaratively. A declarative system expresses the desired state rather than the sequence of steps to achieve it. Kubernetes manifests, Helm values, and Kustomize overlays are all declarative.

Desired State ≠ Imperative Commands
02

Versioned & Immutable

The desired state is stored in a way that enforces immutability and retains a complete version history. Git provides this natively — every commit is a snapshot, every change is audited, and nothing can be silently altered.

Git = Audit Log + Time Machine
03

Pulled Automatically

Software agents (like ArgoCD) automatically pull the desired state declarations from Git and apply them to the system. This pull-based model is more secure than push-based CI/CD because the cluster credentials never leave the cluster.

Pull > Push (security wins)
04

Continuously Reconciled

Software agents continuously observe the actual system state and attempt to apply the desired state. If someone manually edits a resource in the cluster (config drift), the agent detects and corrects it automatically.

Drift Detection → Auto-heal
How It Works

The ArgoCD Workflow

From a code commit to a running deployment — here's how GitOps with ArgoCD works end to end.

1

Developer Pushes Code

A developer merges a pull request. The application source code changes are committed to the app repository.

2

CI Pipeline Builds & Tags Image

A CI system (GitHub Actions, Jenkins, etc.) builds a Docker image, runs tests, and pushes the image to a container registry with a new tag.

3

Config Repo Gets Updated

The CI pipeline (or a bot) updates the image tag in the GitOps config repository — the Kubernetes manifests or Helm values file.

4

ArgoCD Detects the Drift

ArgoCD continuously polls the config repo (or uses a webhook). It detects that the live cluster state no longer matches the desired state in Git.

5

ArgoCD Syncs to Cluster

ArgoCD applies the updated manifests to the Kubernetes cluster — rolling out the new image via the Deployment object.

6

Health Check & Status

ArgoCD monitors rollout health. If pods become unhealthy, it marks the app Degraded and can auto-rollback. The dashboard shows real-time status.

Key Concepts

ArgoCD Concepts

Important terms and objects you'll encounter when working with ArgoCD.

Application CRD

The core ArgoCD custom resource. An Application defines the source (Git repo + path) and destination (cluster + namespace) for a deployment.

AppProject CRD

A grouping mechanism for Applications. Projects restrict which repos and clusters an app can use, enabling multi-team isolation and governance.

Sync Operation

The process of making the live cluster state match the Git desired state. Can be triggered manually or set to auto-sync on every Git commit.

Health Status Status

ArgoCD evaluates resource health: Healthy, Progressing, Degraded, Suspended, or Missing. Health checks are customisable via Lua scripts.

Repository Config

A Git repository registered with ArgoCD. Supports HTTPS and SSH authentication. Can be a dedicated GitOps repo separate from the app source code.

Sync Hooks Lifecycle

Jobs or resources that run at specific points in the sync lifecycle: PreSync (DB migrations), Sync, and PostSync (smoke tests).

Sync Waves Ordering

Use the argocd.argoproj.io/sync-wave annotation to control the order in which resources are applied. Lower wave numbers are applied first.

ApplicationSet CRD

Automates the creation of many Applications from a template. Uses generators (List, Git, Cluster, Matrix) to create apps for multiple environments or clusters dynamically.

Understanding the Difference

GitOps vs Traditional CI/CD

ArgoCD is a CD tool — it complements your CI pipeline rather than replacing it.

Aspect GitOps / ArgoCD Traditional Push CI/CD
Deployment model Pull-based — agent pulls from Git Push-based — pipeline pushes to cluster
Cluster credentials Credentials stay inside the cluster Credentials stored in CI system (risk)
Drift detection Continuous monitoring & auto-heal No detection after deployment
Rollback git revert — instant Re-run pipeline with old tag
Audit trail Git history is the audit log Scattered across CI logs
Source of truth Git repository Running cluster state
Multi-cluster Native support Complex scripting required
